游标

Oracle处理sql语句的时候，会在内存中分一个区，游标相当于指向这个区的句柄或指针。

任何sql语句，服务器都会打开一个游标处理该语句。不过有时候是显式的，有时候是隐式的。

insert update delete 和只有一行的select into语句：隐式游标

对于多行结果的处理，要定义显式游标.

当游标被打开，发生下列情况：

  • 检验绑定变量的值。
  • 基于查询确定活动集。
  • 指针指向活动集第一行。

每个Fetch之后，指针指向下一行。

触发器

三个语句：update、delete、insert

两个时机：before、after

两个级别：行触发或语句触发

行触发器

触发语句处理每一行时，行级别触发器都激发一次。

在触发器的内部，可以通过相关标识符来访问当前被处理行中的数据。即：old和：new 。二者是特殊类型的绑定变量，二者是表的行类型。

:new.field引用行的数据

Insert :old Null :new 新插入的值

Update : old 旧值 :new 新值

Delete :old 旧值 :new null

when子句：只对行触发器有效

语句触发器

一个语句触发器只会针对指定DML语句激活一次。比如，一个update语句，无论update多少行，也只会调用一次update语句触发器。

触发器谓词：

  • Inserting 如果触发语句为insert,则为true
  • Updating 如果触发语句为update,则为true
  • Deleting 如果触发语句为delete,则为true

替代触发器

Instead of 触发器只能定义于视图上，触发事件是视图上的DML，主要用于修改一个本来不能修改的视图。

DML触发器激发顺序

一个表可以拥有定义在其上的任何数量的触发器，可以包括多个DML类型。例如可以定义两个在delete语句级别之后的触发器，相同类型的触发器被激发的顺序没有定义，如果该顺序很重要，那么就把所有的操作结合到一个触发器中。

  1. 如果存在，执行语句级别之前的触发器

  2. 对于受语句影响的每一行：

    1. 如果存在，执行行级别之前的触发器。
    2. 执行语句本身
    3. 如果存在，执行行级别之后的触发器

  3. 如果存在，执行语句级别之后的触发器。

事务

事务是由相关操作构成的一个完整的操作单元，是一系列数据改变的sql语句

在一个事务内，数据的修改一起提交或撤销，如果发生故障或系统错误，整个事务也会自动撤销。提交之后，就不能撤销，提交成功后其他用户才可以通过查询浏览数据的变化。

对数据库所做的一系列修改，在修改过程中，暂时不写入数据库，而是缓存起来，用户在自己的终端可以预览变化，直到全部修改完成，并经过检查确认无误后，一次性提交并写入数据库。

分为隐式和显式两种

  • 隐式：

    隐式提交：DDL（create），DCL（exit，quit，grant，revoke）由系统自动完成提交或撤销，无需用户干预

    隐式回退：异常结束，或系统故障；或系统环境变量autocommit设置为on，那么显示提交的数据也将被自动提交，变为隐式

  • 显式：DML（insert），需要用户操作commit/rollback/savepoint

commit：数据库事务提交，将变化写入数据库，受影响的数据行上的锁被释放。

rollback：回滚语句使数据库回到上次最后提交事务的状态，数据的改变被撤销，之前的数据状态恢复，受影响的数据行上的锁被释放。

savepoint：保存点

Commit and rollback之前的数据状态：

  • 数据可以被恢复.
  • 当前用户可以用select语句查询修改结果.
  • 其他用户不可以查看当前用户对数据做的修改
  • 受影响的数据行被锁; 其他用户不能改变这些行.

何时开始事务：

  • 开始第一个会话（连接数据库）时
  • 执行第一条sql语句时（select不算）

何时结束事务：

  • 执行commit，rollback，DDL
  • 断开与oracle连接（exit），事务自动提交
  • 用户进程异常终止，当前事务回滚

物理结构

database

包括数据文件、控制文件、日志文件，必须的，有多个；其他包括参数文件、口令文件、归档日志文件

数据文件

存储全部数据：表中数据，索引数据

通常后缀.dbf

通过DBWR从数据库缓冲区高速缓存写入到数据文件。

日志文件

redo log files/重做日志文件

记录数据库全部变更，可能是DML、DDL语句。在系统发生故障时，用它对数据库进行恢复

通常名称Redo*.log

LGWR从重做日志缓冲区将重做日志写到日志文件中。

至少有两个日志文件，循环写入

根据在事务信息将被覆盖时，是否应该将文件归档，数据库分为两种归档模式：archivelog（归档日志）（保存历史日志项），noarchivelog（非归档日志）模式

控制文件

control files

oracle打开数据库时需要读取的文件。

是较小的二进制文件，记录了数据库的物理结构：数据库名，数据文件、日志文件的名称和位置等。

通常名称Control*.ctl

至少有一个控制文件，且控制文件的内容只能由oracle本身来修改。控制文件损坏或丢失，会导致无法打开数据库。

参数文件

parameter files 目前已不使用

控制内存、实例

是ASCII文本文件，initsid.ora（名称和内容都不能更改，可以通过其他接口查看），记录数据库运行时的一些重要参数，其中的参数值决定数据库和实例的特性。

服务器参数文件

server parameter files——SPFile

是二进制文件，spfilesid.ora，不能更改

oracle实例启动读取参数文件的顺序：spfilesid.ora->initsid.ora

instance实例

用户访问物理数据库需要借助实例，一个实例通常对应一个数据库。

组成：系统全局区SGA、后台进程

系统全局区SGA

SGA system global area

SGA=数据库缓冲区高速缓存+重做日志缓存+共享池+java池+大池+fixed pool（?）

是实例的主要部分，oracle内存结构的主要组成部分，是oracle为一个实例分配的一组共享内存缓冲区。

保存着oracle系统与所有数据库用户的共享信息，包括共享数据、sql语句分析、重做日志管理等。

把重做日志缓冲区的日志项写到日志文件中，异常中止，数据文件未全部完成时，可通过重做日志文件恢复。

写入时机：

  2. 1/3满
  3. 1MB重做日志（？达到1M吗）

  1. 启动时实例恢复

  2. 释放资源

CKPT

  2. 更新数据文件头
  3. 更新控制文件头

检查点后，物理文件达到一致状态。若不一致，开启数据库后通过SMON恢复。

逻辑结构

逻辑结构支配一个数据库如何使用系统的物理空间。

  • 数据库从逻辑上由一个或多个表空间组成
  • 表空间由段组成
  • 段由一组区组成
  • 区由一组连续的数据块组成
  • 块对应硬盘上的一个或多个物理块。

表空间

和表是完全不同的概念！

逻辑存储结构最高层，一个数据库通常由若干表空间组成，每个表空间可以对应若干数据文件。

表空间中对象的存储：

  • 如果表空间对应一个数据文件，所有对象都存储在此数据文件中；
  • 如果表空间对应多个数据文件，可将一个对象的数据存储在该表空间的任意一个数据文件中，也可一个对象的数据分布在表空间的多个数据文件中。

表空间分类

Oracle中除了用户创建的存放数据对象的数据表空间外，还有以下类型的表空间：

（1）System表空间，存系统数据，包括数据字典。

（2）回滚表空间（RollBack TableSpace）执行后未提交的值，回滚/提交后释放空间

（3）临时表空间（Temp TableSpace）不断在分配，临时的排序数据等

表空间优点

在数据库中使用多个表空间具有如下优势：

（1）能够将数据字典与用户数据分离开来，避免由于字典对象和用户对象保存在同一个数据文件中而产生的I／O冲突。

（2）能够将回滚数据与用户数据分离开来，避免由于硬盘损坏而导致永久性的数据丢失。

（3）能够将表空间的数据文件分散保存到不同硬盘上，平均分布物理I／O操作。

（4）能够将某个表空间设置为脱机状态或联机状态，以便对数据库的一部分进行备份和恢复。

（5）能够将某个表空间设置为只读状态，从而将数据库的一部分设置为只读状态。

（6）能够为某种特殊用途专门设置一个表空间，比如临时表空间等，以优化表空间的使用效率。

（7）能够更加灵活地为用户设置表空间限额。

段

表空间中以段为单位进行分配

段是对象的容器

如：视频、long等大对象，单独放一个段

段分类

1．数据段：保存的是表中的记录。

2．索引段：保存索引中的索引条目。

3．临时段：当处理查询时，Oracle可能会需要使用到一些临时存储空间，用于临时保存解析过的查询语句以及在排序过程中产生的临时数据。

4．回滚段：用于存放数据修改之前的值（包括数据修改之前的位置和值）。

区

段由区组成，每个段在创建时都会分配一定数目的初始区（主区），当主区用完，自动为段分配新区。

区的大小可能不同，可以是一个数据块，也可以达到2G

区是分配的单位

数据库块

块是最小的空间分配单位。

数据行、索引条目、临时排序结果存储在块中。通常oracle 从磁盘读写的就是块。

常见大小：2KB 4K 8K 16K 32K，也可以自行设置

数据字典

汇集了数据库运行所需要的基础信息，是oracle数据库的核心组件，随着数据库的产生而产生，随数据库变化而变化。

数据字典由sys拥有，在system表空间中。

包括：

  • 数据库所有对象的定义（表、视图、索引、群集、同义词、序列、过程、函数、包、触发器等）
  • 为对象如何分配和使用空间
  • 列的默认值
  • 完整性约束信息
  • 数据库安全信息，包括用户、权限、角色等
  • 审计信息

数据字典的结构

基础表

存储相关数据库信息的表，包括表、索引、约束，以及所有其他数据库对象结构的信息，是oracle系统的核心。system表空间的表非常重要，应避免以sys登录破坏基表。

数据字典视图

USER视图:前缀为USER，当前用户模式下所有对象的信息。

Oracle维护了一种从数据库内存结构构建的虚拟表，记录当前数据库的活动。这些表称为动态性能表。在这些表上面创建的视图被称为动态性能视图（dynamic performance view）。

备份与恢复

数据库备份恢复是保证数据库安全运行的一项重要内容。

是数据库管理员的重要职责。

由于数据库的物理结构被误操作、机器硬件故障而遭到破坏，必须使用数据库的备份文件对数据库实施恢复，尽可能使用户的数据免遭损失，使数据库运行正常。

所谓备份 ，就是把数据库复制到转储设备的过程。其中，转储设备是指用于放置数据库拷贝的磁带或磁盘。

逻辑备份

逻辑备份是利用SQL语言从数据库中抽取数据并存于二进制文件的过程,生成文件后缀为dmp。

用导出/导入命令(Export/Import-exp/imp)导出是数据库的逻辑备份，导入是数据库的逻辑恢复。

数据库逻辑备份是物理备份的补充，适用于数据较少时，使用灵活

Oracle支持三种类型的导出:

(1)表方式，将指定表的数据导出。

(2)用户方式，将指定用户的所有对象及数据导出。

(3)全库方式，将数据库中的所有对象导出。

数据库必须处于打开状态

exp

imp

Import读取由Export导出的二进制转储文件并执行其中的命令。

可以导入所有或部分导出数据。

可以在IMPORT时使用FROM USER和TO USER参数来确定要导入的USER，但是要保证TO USER一定已经存在。

不加 FROMUSER和TOUSER,默认原先是谁的对象还是导给谁。

物理备份

物理备份是将实际组成数据库的操作系统文件从一处拷贝到另一处的备份过程，它涉及到组成数据库的文件，但不考虑其逻辑内容。物理备份包括脱机备份（冷备份）和联机备份（热备份）两种方式。

脱机备份

主要指在正常关闭数据库的状态下进行的数据库完全备份，备份内容包括所有数据文件、控制文件、联机日志文件及参数文件（spfile）。（D:\oracle\product\10.2.0\db_1\dbs\SPFILEORCL.ORA）

脱机备份必须在数据库已经正常关闭的情况下进行，此时，系统会提供给用户一个完整的数据库

优点：

（1）是非常快速的备份方法（只需拷贝文件）。

（2）容易恢复到某个时间点上（只需将文件再拷贝回去）。

（3）能与归档方法相结合，作数据库“最新状态”的恢复。

（4）低度维护，高度安全。最快和最安全的方法

缺点：

(1)单独使用时，只能提供到“某一时间点上”的恢复。

(2)在实施备份的过程中．数据库必须备份而不能做其他工作。也就是说，在冷备份过程中，数据库必须是关闭状态。

(3)若磁盘空间有限，只能拷贝到磁带等其他外部存储设备上，速度会很慢。

联机备份

完全数据库脱机备份不足之处：备份期间要关闭数据库，影响用户正常使用。

联机备份也叫热备份，是在数据库正常运行时进行，备份期间不影响用户使用。

数据库联机备份包括控制文件、数据文件、归档日志文件备份等。

联机备份由于数据库文件之间存在不同步，在备份拷回数据库时需要实施恢复。所以只能在归档模式下使用。

由于联机备份需要消耗较多的系统资源，因此DBA应安排在数据库不使用或使用率较低的情况下进行。

优点：

（1）可在表空间或数据文件级备份，备份时间短。

（2）备份时数据库仍可使用。

（3）可达到秒级恢复（恢复到某一时间点上）。

（4）可对几乎所有数据库实体作恢复。

（5）恢复是快速的，在大多数情况下在数据库仍工作时恢复。

缺点：

（1）不能出错，否则后果严重。

（2）若联机备份不成功，所得结果不可用于时间点的恢复。

（3）较难维护，必须仔细小心，不允许失败。

数据库恢复

数据库恢复，就是当发生故障后，利用已备份的文件，重新建立一个完整的数据库。根据出现故障的原因，恢复分为两种类型：

  • 实例恢复：这种恢复是Oracle实例出现失败后（意外掉电，后台进程故障），Oracle自动进行的恢复。
  • 介质恢复：针对介质故障（一个文件、一个文件的一部分或一个磁盘不能读或不能写）这种恢复是当存放数据库的介质出现故障时所做的恢复。

在归档方式下，根据数据库的恢复程度，将恢复方法分为两种类型：

完全恢复：将数据库恢复到数据库失败时数据库的状态。这种恢复是通过装载数据库备份并应用全部的重做日志做到的。

不完全恢复：将数据库恢复到数据库失败前的某一时刻数据库的状态。这种恢复是通过装载数据库备份并应用部分的重做日志做到的，进行不完全恢复后须在启动数据库时用 resetlogs 选项重设联机重做日志。
